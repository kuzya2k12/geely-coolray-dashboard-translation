# Формат RLAD (Infineon Run-Length Adaptive Dithering)

Формат сжатия графики, применяемый в приборке Coolray/Binyue L (Infineon TRAVEO II).
В норме декодируется аппаратно графическим движком чипа; публичного софт-декодера
не существовало. Формат ниже **восстановлен методом «оракула»** (референс-энкодер
Infineon кодирует контролируемые входы, выход разбирается) и **проверен**
round-trip-тестом `decode(encode(x)) == x`.

Реализация декодера: [`geely_cluster/rlad_decoder.py`](../geely_cluster/rlad_decoder.py).

---

## Структура RLAD-пакета (RGBA8888, 8 пикселей на пакет)

Пакет описывает 8 пикселей. Начинается с 6-байтного заголовка, затем идёт битовый
поток значений:

```
байт 0:  младший ниббл = cbpc[R], старший ниббл = cbpc[G]   (бит на значение канала)
байт 1:  младший ниббл = cbpc[B], старший ниббл = cbpc[A]
байт 2:  bias[R]   (базовое значение канала R)
байт 3:  bias[G]
байт 4:  bias[B]
байт 5:  bias[A]

далее — для каждого канала c, у которого cbpc[c] > 0:
    8 значений по cbpc[c] бит каждое.
    итоговое значение пикселя канала = bias[c] + offset
```

- `cbpc` (compressed bits per channel) — сколько бит занимает одно значение канала.
  Если `cbpc[c] == 0` — весь канал постоянен и равен `bias[c]` (данные не пишутся).
- Биты читаются из 32-битных слов little-endian, порядок бит **lsb → msb**
  (как в `RLAD::BitStream` из SDK Infineon).

## Структура RLAD-пакета (A8 — формат ГЛИФОВ ШРИФТА)

A8 — одноканальный alpha-формат (8 бит). Именно в нём хранятся **глифы шрифта**
приборки. Упакован иначе, чем RGBA8888 (один канал вместо четырёх). Восстановлен
методом oracle (`ResourceGenerator.exe -foA8 -cRLAD`) и проверен на тест-глифе.

Пакет = до 8 пикселей:
```
4 бита  : cbpc      (бит на значение alpha; 0 = все значения = bias)
8 бит   : bias      (базовое значение alpha)
N × cbpc: offset'ы  (значение alpha = bias + offset), только если cbpc > 0
          где N = число пикселей в пакете (обычно 8)
```
Тот же битовый поток (32-битные слова LE, lsb → msb).

**Упаковка ПОСТРОЧНАЯ (важно, подтверждено oracle-диффом):** каждая строка
изображения начинается с НОВОГО пакета; пакеты **НЕ переходят границу строки**.
Если ширина не кратна 8, последний пакет строки покрывает остаток `width % 8`
пикселей (N < 8). Ранняя версия декодера читала поток непрерывно и «съезжала» на
ширинах, не кратных 8 — это и была недостающая деталь формата.

**Флаг сжатия:** байт `flags` (+11 в 20-байтном заголовке) определяет режим.
`flags = 0x00` → данные НЕ сжаты (сырые байты alpha, по строкам). `flags != 0`
(напр. 0x05) → RLAD-поток по схеме выше. ResourceGenerator сам выбирает несжатый
режим, если RLAD невыгоден (мелкие/пёстрые изображения). Декодер должен смотреть
на `flags`.

Реализация: [`decode_a8_image`](../geely_cluster/rlad_decoder.py). Проверено тестами
[`tests/test_rlad_decoder.py`](../tests/test_rlad_decoder.py): `a8:glyph` (16×16,
кратная) и `a8:rowwise` (12×12, ширина НЕ кратна 8 — проверка построчности).

### cbpc = 8 в A8 — это lossless (не lossy!)
Важное уточнение: в RGBA8888-декодере cbpc=8 включал lossy-dithering (расхождение).
В **A8 такого нет** — cbpc=8 означает просто 8 бит на значение alpha напрямую
(bias=0, offset=само значение), декодируется точно. Проверено oracle-диффом на
входах с максимальным разбросом (`[0,255,0,255,...]`) — 0 расхождений. Т.е. формат
глифов (A8) декодируется **на 100% без потерь во всех режимах cbpc 0..8**.

## Режимы (по SDK Infineon)

`CYGFX_SM_COMP`: `RLAD=0x0`, `RLAD_UNIFORM=0x1`, `RLA=0x2`, `RLC=0x3`, `NONE=0x4`.
Константы RLAD: 8 пикселей/пакет, 4 канала, максимум 8 бит на канал.
Порядок байт: RL — big-endian; RLA/RLAD — little-endian.

## Пиксельные форматы (`CYGFX_SM_FORMAT`)

`0=RGBA8888` … `16=R5G6B5`, `17=A8RGB8` (gray/indexed), `18=RGB8` (8bpp gray/idx),
`19=A8`, `20=A4RGB4`, `21=A4`, `22=A2`, `23=A1`, `24=RGB1` (1bpp ч/б).

Глифы шрифта — обычно **A8/A4** (alpha-битмапы). У них плавные края → небольшой
`cbpc` → это надёжно работающий режим декодера.

## Сигнатуры начала (из выхода референс-энкодера)

Начало пакета для белого-на-прозрачном тестового глифа:
- RLAD: `00 80 FF FF FF ...`
- RLA:  `XX 00 C4 FF FF FF ...`
- RL:   `FF FF FF 81 ...`

(На реальных данных приборки bias/значения другие — это лишь ориентир формата.)

## Точность декодера

- **Работает 100 %** на: сплошной цвет, линейный ramp, ненулевой bias,
  alpha-градиенты, сглаженные края глифов — т.е. режим малого/среднего `cbpc`.
  Это именно то, из чего состоят глифы шрифта.
- **Расхождение** только при `cbpc = 8` (максимальный разброс, напр. чередование
  0/255) — там включается lossy-dithering и порядок offset иной. Для текста/глифов
  (плавная alpha) этот режим не встречается, поэтому на локализацию не влияет.

## Surface-дескриптор Infineon (20 байт)

Восстановлен из функции `utSurfLoadBitmapEx` библиотеки Infineon:

```
+0x00  u32  nSize        полный размер записи (данные = nSize - 0x14)
+0x04  u16  width
+0x06  u16  height
+0x08  u16  stride       байт на строку
+0x0A  u8   bitperpixel
+0x0B  u8   flags        биты формата/сжатия (проверяются &5, &0x1e; старший = палитра)
+0x0C  u32  colorbits    ширины каналов R,G,B,A по нибблам (маска 0x0F0F0F0F)
+0x10  u32  colorshift
+0x14  ...  данные (или заголовок палитры, если flags & палитра)
```

Реализация парсера: [`geely_cluster/surface.py`](../geely_cluster/surface.py).

> Важно: в дампах приборки Coolray A5 **отдельной таблицы этих дескрипторов не
> найдено** — она в защищённой flash MCU. Поэтому дескриптор задокументирован как
> формат, но «навести» его на данные без таблицы нельзя (см. INVESTIGATION_HISTORY.md).
