# История исследования: гипотезы, тупики и неуспешные пути

Этот файл — честная «археология» проекта. Здесь собраны **неподтверждённые гипотезы,
отвергнутые версии и методы, которые НЕ сработали**. Цель — чтобы следующий человек
не тратил время на уже пройденные тупики и понимал, как менялось понимание.

Достоверные факты и рабочий результат — в [README](../README.md) и
[RLAD_FORMAT.md](RLAD_FORMAT.md). Здесь — то, что осталось за кадром.

---

## 1. Отвергнутые версии (что думали → что оказалось)

### «Алжирский эксплойт по CAN / U-Boot / buffer overflow»
Стартовая легенда сообщества: некий разработчик нашёл уязвимость в приборке, эксплойт
по CAN-шине, отладочная консоль U-Boot, переполнение буфера, root, правка `/dev/block`,
заливка по OBD2.
→ **Не подтверждено ничем.** «U-Boot-консоль по шине» — несуществующая техника.
Автора-первооткрывателя в открытых источниках нет. Похоже на правдоподобную
реконструкцию (в стиле генерации LLM), а не описание реального метода. Реальные
локализации делаются правкой ресурсов, а не «эксплойтами».

### «UI приборки лежит в другом чипе / скрытом eMMC» — ОТОЗВАНО
Когда выяснилось, что 2×64 МБ = мало для «полноценной» приборки, возникла версия, что
основной UI (Linux + Kanzi) лежит в отдельном большом eMMC, который мы не сняли.
→ **Неверно.** TRAVEO II CYT3DL — это микроконтроллер (Cortex-M7, без MMU под
полноценную ОС). Linux он не тянет, значит и гигабайтного eMMC под Linux нет.
Правильный вывод: приборка A5 — **самодостаточный MCU-дизайн**, всё умещается в
128 МБ NOR + 4 МБ внутренней flash. Малый размер — это не «сняли не всё», а
принципиально другая, минимальная архитектура (см. README).

### «У нас Kanzi/.kzb, как на других Geely» — НЕ для нашей платформы
Владельцы родственных моделей (SX11-A3 / Boyue / Preface) сообщают, что их приборки
работают под QNX/Linux, а UI сделан в **Kanzi Studio** и лежит в файлах `.kzb`
(напр. `app/rc_hmi/sx11.kzb`), которые редактируемы.
→ Для **нашей SX11-A5 не подтверждено и, скорее всего, неприменимо.** В наших
NOR-дампах поиском сигнатур НЕ найдено ни `.kzb`, ни файловой системы, ни QNX/Linux —
только RLAD-данные. Это платформенная разница (см. раздел «CMA vs BMA» ниже). Kanzi
оставлен как справка про соседние платформы, не как план для A5.

## 2. Методы поиска глифов/таблицы, которые НЕ сработали

Все перечисленные дали 0 или шум. Причина одна: **таблицы ресурсов нет в NOR-дампах**
(она в защищённой flash MCU), а «слепой» разбор без неё невозможен.

1. **Поиск surface-дескрипторов Infineon** (20-байтный заголовок) сканированием — 0
   валидных при всех вариантах stride.
2. **Поиск таблицы указателей SMIF** (значения вида `0x60xxxxxx`) — только статистический
   шум, связной таблицы нет.
3. **Строгая сигнатура A8-глифа** (alpha-only, bias 0/255, смесь пустых и штриховых
   пакетов) — 0 совпадений в дампе.
4. **Сырой A4/A8-атлас** (перебор ширины, детекция сетки глифов) — найденная
   «регулярность» оказалась границами блоков данных + шумом между ними, не глифами.
5. **Слепой RLAD-декод с произвольных смещений** — шум и ложные срабатывания (декодер
   «успешно» разжимает случайные байты в мелкие значения; визуально — не картинки).
6. **Побайтовый поиск китайского текста** (GB2312/GBK/UTF-16), даже с фильтром по
   частотным «приборочным» иероглифам — десятки тысяч совпадений, но все = шум сжатых
   данных, осмысленных слов ноль. Подтверждает: текст хранится как индексы глифов.
7. **Поиск таблицы адресов секций** — взяты реальные границы блоков (по скачкам
   энтропии), искались как uint32 (абсолютные / SMIF-база / номер сектора); связной
   таблицы нет, 98 «кандидатов» оказались повторами байт внутри данных.
8. **Метод FTools** (смещения как абсолютные адреса от фикс-базы, как в Ford IPC) —
   перебор баз `0x60000000/0x08000000/...`; только ложные монотонные
   последовательности, настоящей таблицы нет.
9. **Гипотеза «A5A5A5A5 = контейнер Infineon ResourceGenerator»** — ПРОВЕРЕНА и
   ОПРОВЕРГНУТА. Прогнали официальный `ResourceGenerator.exe` (из SDK Infineon) во всех
   режимах: обычный бинарь (`-b`), append несколько ресурсов (`-a`), partitioning (`-p`).
   Ни в одном режиме `A5A5A5A5` не появляется — вывод всегда начинается с заголовка
   ресурса (nSize, w, h, stride, bpp, flags, colorbits). Значит `A5A5A5A5` в нашем
   дампе — **обёртка/раздел уровня Geely, а не формат Infineon**. Оракул-подход к
   контейнеру (как сработал с RLAD-пикселями) здесь не проходит — контейнер закрытый.
10. **Слепой A8-декод дампа + строгий A8-детектор глифов** — ПРОВЕРЕНО, шум. После
    дореверса A8 (формат глифов) прогнали A8-декодер по всему дампу со строгим
    детектором (валидный cbpc, смесь пустых/чернильных пакетов, доминирующий фон,
    мало уровней alpha). Детектор дал ~112 «кандидатов», НО **визуальная проверка
    всех отрендеренных картинок — снова шум** (серые блоки/полосы, ни одного глифа).
    Причина та же: без таблицы ресурсов декодер стартует не с начала глифа и не знает
    ширину; любой RLAD-подобный поток из случайных байт «разжимается» в фон+шум.
    **Перепроверено после починки построчного декодера, на ОБОИХ чипах (U17 и U18),
    с перебором ширин 12–64 и визуальным просмотром десятков PNG — результат тот же
    шум (горизонтальные полосы = границы блоков).** Метрики (в т.ч. score=1.000)
    оказались ложными: идеальный балл давали ОДНОРОДНЫЕ/пустые зоны, а не глифы.
    Вывод окончательный: даже с идеально работающим A8-декодером слепой поиск в дампах
    невозможен — барьер не в декодере (он верен, тесты 8/8), а в отсутствии привязки
    (таблицы ресурсов). Шумовые PNG-примеры в репозиторий НЕ включены как бесполезные.
11. **Метод «цепочка заголовков» (из анализа SDK Infineon)** — ПРОВЕРЕНО, не найдено.
    Ключевой инсайт из разбора репо Infineon: оглавления-таблицы НЕ существует ни у
    кого — ресурсы просто склеены встык, а «индекс» = хардкод-указатели в коде MCU.
    Правило обхода: найти 20-байт заголовок → прыгнуть на `off + nSize` → там должен
    быть следующий валидный заголовок; цепочка = карта всех ресурсов без таблицы.
    Прогнали по нашему дампу со строгой валидацией (nSize/w/h согласованы,
    `stride == ceil(w*bpp/8)`): **0 цепочек длиной ≥5**. Дополнительно: 0 мест с ТОЧНЫМ
    stride, а обязательное поле RGBA-заголовка `ColorComponentShift = 18 10 08 00`
    встречается лишь 2 раза на 64 МБ (в реальном surface-контейнере были бы сотни).
    Вывод: **стандартных Infineon-контейнеров (ResourceGenerator) в нашем дампе НЕТ.**
    Формат Geely — иной. См. docs/INFINEON_REPOS.md (что даёт открытая экосистема, а
    чего в ней принципиально нет).
12. **Служебные зоны NOR (boot-header / partition table / metadata)** — ПРОВЕРЕНО, пусто.
    По даташиту S25HL512T мелкие 4КБ-сектора (типичное место метаданных) — в начале ИЛИ
    конце чипа; агент рекомендовал искать таблицу в первых и последних 128КБ. Проверили:
    - Начало дампа: только заголовок `A5A5A5A5` + одно 16-бит значение (U17=0x20E0,
      U18=0xE2F0), далее **чистые нули до 0x180000**. Это НЕ таблица разделов — адресов
      ресурсов там нет.
    - Последние 128КБ (top-сектора): **полностью пусты** (0 значимых байт).
    - Значение из заголовка (0x20E0/0xE2F0) — разное у чипов, к границам данных не
      привязывается (проверены множители ×1/×64/×256/×512/×4КБ). Вероятно версия/CRC.
    - Данные обоих чипов идут до ~0x3450000 (≈52 из 64 МБ используется).
    Вывод: таблицы ресурсов в служебных зонах NOR нет — она в прошивке MCU (хардкод
    указатели в область 0x60000000), а MCU-flash под HSM. См. docs/INFINEON_REPOS.md.
13. **Извлечь готовый декодер `RLAD::Decode` из библиотек Infineon** — НЕВОЗМОЖНО.
    Символов `RLAD::Decode`/`BitStream`/`Package` в `libutil_cm7.a` и
    `libbasic_graphics_cm7.a` нет (реализация в закрытом `03_build`, не в публичных `.a`).
    Публичен только заголовок-интерфейс — по нему декодер и реализован в этом проекте.
14. **Дамп внутренней flash MCU через SWD/JTAG** — ПОДТВЕРЖДЁННО ЗАБЛОКИРОВАН (не просто
    гипотеза). На форуме Infineon есть тема ровно про наш MCU CYT3DLABHBQ1AES: серийную
    (product) плату не удаётся прошить/читать из-за **NAR/SAR** (Normal/Secure Access
    Restrictions) — debug access port CM0+ заблокирован; lifecycle = SECURE. Тот же код
    при этом шьётся на eval-плате без ограничений. Приборка Geely — серийное SECURE-
    устройство → чтение внутренней flash MCU (где таблица ресурсов) через отладку
    практически невозможно. Тема:
    community.infineon.com/t5/TRAVEO-T2G/Flashing-failed-for-CYT3DLABHBQ1AES-product-development-board/td-p/1010153
    ВАЖНО: NAR/SAR ограничивают доступ к MCU, но НЕ к внешнему NOR — его мы читаем
    программатором (дампы есть). Детали — docs/INFINEON_REPOS.md.
15. **Готовые инструменты Infineon для декода bin→картинка** — НЕ СУЩЕСТВУЮТ публично.
    Вся публичная экосистема односторонняя (image→bin), декод делает аппаратный GPU.
    ResourceGenerator/rlad-encoder — только кодеры; Bin2Text — bin→C-текст (не картинка);
    png_Read — только чтение PNG. Kanzi/Candera/Qt формат Infineon не декодируют.
    Поэтому наш `rlad_decoder.py` (oracle-реконструкция) — единственный рабочий софт-декодер.

**НЕПРОЙДЕННАЯ зацепка (gated, но бесплатная): исходники в `03_build` через myICP.**
readme ResourceGenerator: *"The Resource Generator source code is available in 03_build
folder"* — но на GitHub эту папку вырезали (полная поставка = 01_/02_/03_build/04_sample).
`03_build` лежит в полном пакете "Graphics Driver for TRAVEO T2G cluster" на **myInfineon
Collaboration Platform (myICP)**: бесплатный аккаунт myInfineon + запрос доступа на
**traveo@infineon.com**. Там — C-исходник `storeDumpSurfaceBin` (писатель контейнера,
инвертируется в декодер) и исходник RLAD-кодека (энкодер + вероятный ссылочный декодер).
Даёт точный формат/декодер, НО не решает таблицу ресурсов (в закрытом MCU). Материалы
под EULA Infineon — в этот репозиторий НЕ выкладывать, только использовать для своего
кода. Единственный непройденный ход на фронте инструментов. Детали — docs/INFINEON_REPOS.md.

**Про экспортированные изображения:** во время экспериментов рендерились PNG/PPM из
«кандидатных» регионов (и RGBA, и A8). Все они — **визуализация шума**, а не реальные
глифы приборки, что подтверждено визуально. Не принимать их за находки. В репозиторий
не включены как бесполезные.

### РЕШЕНО ПОЛНОСТЬЮ: A8-RLAD дореверсен и верифицирован (формат глифов шрифта)
Изначально декодер был только под **RGBA8888**, A8-вывод ResourceGenerator не читал.
Дореверсили A8 oracle-методом (`ResourceGenerator.exe -foA8 -cRLAD`). Формат A8-пакета:
**4 бита cbpc + 8 бит bias + N×cbpc offset** (один канал; N — число пикселей в пакете).

Три уточнения, найденные при верификации (превратили «работает на глифах» в «формат
понят на 100%»):
1. **Упаковка ПОСТРОЧНАЯ.** Пакеты не переходят границу строки; последний пакет строки
   = остаток `width%8`. Раньше декодер читал непрерывно и съезжал на ширинах, не
   кратных 8 (поймано oracle-диффом на 50×40/37×29). Исправлено.
2. **cbpc=8 в A8 — lossless** (в отличие от RGBA, где cbpc=8 давал lossy-dithering).
   Проверено на `[0,255,0,255,...]` — 0 расхождений. lossy-ветки в A8 НЕТ.
3. **Флаг сжатия** (`flags` @+11): 0x00 = несжато (сырые байты), !=0 = RLAD. Декодер
   смотрит на flags.

Верификация: round-trip decode(ResourceGenerator(x))==x на размерах 16×16, 64×64,
50×40, 37×29, 12×12, 13×20, 7×7 (кратные и НЕ кратные 8; сжатые и несжатые) — 0
расхождений. Тесты `a8:glyph` + `a8:rowwise` в `tests/` (8/8 passed). **Формат глифов
приборки (A8) теперь декодируется полностью и без потерь.**

## 3. Опыт коллег с ДРУГИХ авто (что переносимо, что нет)

Изучены реальные успешные локализации приборок. Общая закономерность:
**никто не переводил приборку из голого чип-офф дампа — работали с update-файлом или
дампом с уже готовым переводом**, где ресурсы и таблица лежат вместе.

- **Ford Focus IPC — `github.com/bigunclemax/FTools`** (открытый C++). Ближайший аналог
  по структуре: таблица смещений + атлас + сжатие + CRC. Приём: смещения = абсолютные
  адреса от фикс-базы (`0x02400000`), файл-offset = addr − base. Работали с VBF
  update-файлом, пересчитывали CRC при пересборке. (У нас метод абс-адресов таблицу не
  нашёл — см. п.8 выше.)
- **Chery Tiggo 8 Pro** (кейс на drive2). QNX-приборка. Автор написал свой
  распаковщик/упаковщик update-файла; первая сборка окирпичила приборку → восстановил
  рефлешем. Стратегически самый близкий: получил **прошивку** (update, не чип-офф) →
  распаковал → правил.
- **Mustang S550 — `Deathstars/mustang-s550-ipc-hebrew`**. Там шрифт = обычный
  встроенный TTF (ищется по magic `00 01 00 00`). У нас стандартных TTF нет — глифы в
  RLAD-битмапах.
- **Предупреждение (Ford i.MX8, Kanzi .kzb)**: даже полный внешний дамп не помог —
  `.kzb` там «не реверсится». У нас в этом смысле повезло: формат RLAD мы вскрыли.

## 4. CMA vs BMA — почему приборки Geely разные

Ключевое разделение (со слов эксперта сообщества):

- **CMA (Monjaro, Preface и др.)** — приборка на прикладном процессоре: Linux + Kanzi
  (или новее Unity) + eMMC (дамп ~8 ГБ). Правится **пофайлово**, перевод относительно
  лёгкий. Все скриншоты `.kzb` с читаемыми строками и «kzb-метод» — отсюда.
- **BMA (наш Coolray / Binyue L, SX11-A5)** — голый микроконтроллер TRAVEO II + 128 МБ
  NOR, графика RLAD, прошивка целиком чипа. Пофайловой замены нет. Это и есть причина,
  почему A5 — самая сложная приборка Geely: **крупным игрокам рынка автопереводов её
  перевести не удалось**; единственная известная успешная локализация — у энтузиастов
  из Алжира в коллаборации с одной СНГ-командой (метод не раскрыт).

Вывод: методы с CMA/Kanzi **напрямую не переносятся** на нашу A5.

### Нюансы формата .kzb (справка, если вдруг пригодится на CMA-платформе)
- `.kzb` — архив **без сжатия**; правится скриптом (распаковать→изменить→запаковать)
  или в hex.
- **Критично:** строка-замена должна быть **строго той же длины**, что оригинал, иначе
  структура разваливается (отсюда сокращения в переводах).
- Строки лежат читаемым текстом; есть секция `Locales` (напр. en-US/neutral/zh-CN).
- Инструменты: **KZB Explorer** (GUI), **python-kzb** (распаковка; запаковка не
  проверена сообществом, часть картинок выходит битой).

## 5. Про ADB и дамп ГУ (частая путаница)

- **ADB к приборке невозможен**: приборка — микроконтроллер Infineon, не Android.
  Дамп её внутренней flash — только SWD/JTAG (и почти наверняка заблокирован HSM).
- **ГУ (большой экран, Flyme/Android) и приборка — РАЗНЫЕ устройства.** К ГУ ADB
  подключается, дамп ГУ снять можно, но **ресурсов приборки в дампе ГУ нет**.
- Единственная возможная польза дампа ГУ для нашей задачи — поискать в Android-части,
  **как ГУ общается с приборкой по CAN** (CarService/VHAL, CAN-обработчики, возможный
  механизм обновления приборки). Связь ГУ↔приборка идёт по CAN (по словам разработчика
  сообщества: медиа-текст на приборку шлётся по CAN, обложки альбомов «не влезают в
  CAN-кадр»).

## 6. Что использовалось для разбора (инструменты)

- **Референс-энкодер Infineon RLAD** (`rlad-encoder.exe`, из открытого примера
  ModusToolbox `mtb-example-psoc-edge-gfx-rlad`) — как «оракул»: кодируем известные
  картинки, разбираем выход. Запускается на Windows. В репозиторий не включён
  (проприетарный) — качается из SDK Infineon.
- **capstone** (pip) — дизассемблер ARM Thumb; из библиотек Infineon
  (`libutil_cm7.a`) восстановлены формат RLD и структура surface-дескриптора.
- **Pillow** (pip) — рендер/экспорт изображений.
- Клон `Infineon/tviic2d-gfx-mw` — заголовки форматов (истина по структурам). В

## Поиск фона/крупных ассетов и структурный поиск (что сделано — БЕЗ выводов)

Что проделано (результаты предварительные, алгоритм может быть неверен — выводы рано):
- **Поиск больших плавных регионов (кандидаты в ФОН).** Идея: фон приборки — крупный
  градиент; переключается тёмная/светлая тема, значит фонов несколько.
  - RGBA-декод + метрика «плавность» → 0 кандидатов с высоким score.
  - Чтение СЫРЬЁМ как RGB565/RGBA8888 + метрика плавности → ~30 регионов score≈1.0.
    При визуальной проверке: часть — почти однотонная заливка (напр. зона 0x344xxxx,
    почти белая — возможно светлая тема ИЛИ просто заливка); часть (0x0088E000,
    0x028FA000 и др.) — цветные РОВНЫЕ полосы сверху + цветной «шум» ниже.
  - Наблюдение (НЕ вывод): ровные цветные полосы похожи на несжатые куски/заголовки,
    «шум» ниже — вероятно RLAD-сжатое тело. Т.е. в дампе ЕСТЬ цветные RGB565-данные,
    но крупные ассеты, судя по всему, сжаты и сырьём не читаются целиком.
  - score=1.0 у метрики оказывался ложным на ОДНОРОДНЫХ зонах — метрике не доверять.
- **Итог сессии:** узнаваемого цельного фона получить НЕ удалось; это не опровергает и
  не подтверждает алгоритм — привязки (начало+ширина+формат сжатого региона) по-прежнему
  нет. Шумовые/полосатые PNG в репозиторий не включены.
