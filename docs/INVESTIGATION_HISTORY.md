# История исследования: гипотезы, тупики и неуспешные пути

Этот файл — честная «археология» проекта. Здесь собраны **неподтверждённые гипотезы,
отвергнутые версии и методы, которые НЕ сработали**. Цель — чтобы следующий человек
не тратил время на уже пройденные тупики и понимал, как менялось понимание.

Достоверные факты и рабочий результат — в [README](../README.md) и
[RLAD_FORMAT.md](RLAD_FORMAT.md). Здесь — то, что осталось за кадром.

## 1. Отвергнутые версии (что думали → что оказалось)

### «Алжирский эксплойт по CAN / U-Boot / buffer overflow» — легенда только про МЕХАНИЗМ
**Важно разделить факт и легенду.**

**Факт (подтверждён):** энтузиасты из Алжира РЕАЛЬНО ставят перевод приборки, причём,
судя по всему, **без физического разбора и чип-оффа**. Это не миф — переводы существуют
(см. фото `photos/not-original-french-translation-example*`). Каким именно способом они
заливают перевод — **неизвестно** (варианты: CAN через ГУ, диагностика через ELM/OBD2,
инженерное/сервисное меню, свой софт). Метод они не раскрывают.

**Легенда (не подтверждена):** конкретное ТЕХНИЧЕСКОЕ описание, которое гуляло по
сообществу — «эксплойт по CAN + отладочная консоль U-Boot + переполнение буфера + root +
правка `/dev/block` + заливка по OBD2».
→ Именно эта формулировка **ничем не подтверждена**. «U-Boot-консоль по шине» —
несуществующая техника (U-Boot — загрузчик Linux, а у нас голый MCU без Linux, см. ниже).
Похоже на правдоподобную реконструкцию, а не на описание реального процесса.

**Вывод:** тот факт, что перевод без разбора возможен на этой платформе — реальный и
важный (значит есть некий рабочий канал заливки). Недостоверно лишь приписанное ему
«хакерское» объяснение. Реальный механизм алжирцев остаётся открытым вопросом и был бы
ценной зацепкой, если кто-то его выяснит.

### «UI приборки лежит в другом чипе / скрытом eMMC» — ОТОЗВАНО
Когда выяснилось, что 2×64 МБ = мало для «полноценной» приборки, возникла версия, что
основной UI (Linux + Kanzi) лежит в отдельном большом eMMC, который мы не сняли.
→ **Неверно.** TRAVEO II CYT3DL — это микроконтроллер (Cortex-M7, без MMU под
полноценную ОС). Linux он не тянет, значит и гигабайтного eMMC под Linux нет.
Правильный вывод: приборка A5 — **самодостаточный MCU-дизайн**, всё умещается в
128 МБ NOR + 4 МБ внутренней flash. Малый размер — это не «сняли не всё», а
принципиально другая, минимальная архитектура (см. README).

### «У нас Kanzi/.kzb, как на других Geely» — НЕ для нашей платформы
Владельцы родственных моделей (SX11-A3 / Boyue / Preface) сообщают, что их приборки
работают под QNX/Linux, а UI сделан в **Kanzi Studio** и лежит в файлах `.kzb`
(напр. `app/rc_hmi/sx11.kzb`), которые редактируемы.
→ Для **нашей SX11-A5 не подтверждено и, скорее всего, неприменимо.** В наших
NOR-дампах поиском сигнатур НЕ найдено ни `.kzb`, ни файловой системы, ни QNX/Linux —
только RLAD-данные. Это платформенная разница (см. раздел «CMA vs BMA» ниже). Kanzi
оставлен как справка про соседние платформы, не как план для A5.

## 2. Методы поиска глифов/таблицы, которые НЕ сработали

Все перечисленные дали 0 или шум. Причина одна: **таблицы ресурсов нет в NOR-дампах**
(она в защищённой flash MCU), а «слепой» разбор без неё невозможен.

1. **Поиск surface-дескрипторов Infineon** (20-байтный заголовок) сканированием — 0
   валидных при всех вариантах stride.
2. **Поиск таблицы указателей SMIF** (значения вида `0x60xxxxxx`) — только статистический
   шум, связной таблицы нет.
3. **Строгая сигнатура A8-глифа** (alpha-only, bias 0/255, смесь пустых и штриховых
   пакетов) — 0 совпадений в дампе.
4. **Сырой A4/A8-атлас** (перебор ширины, детекция сетки глифов) — найденная
   «регулярность» оказалась границами блоков данных + шумом между ними, не глифами.
5. **Слепой RLAD-декод с произвольных смещений** — шум и ложные срабатывания (декодер
   «успешно» разжимает случайные байты в мелкие значения; визуально — не картинки).
6. **Побайтовый поиск китайского текста** (GB2312/GBK/UTF-16), даже с фильтром по
   частотным «приборочным» иероглифам — десятки тысяч совпадений, но все = шум сжатых
   данных, осмысленных слов ноль. Подтверждает: текст хранится как индексы глифов.
7. **Поиск таблицы адресов секций** — взяты реальные границы блоков (по скачкам
   энтропии), искались как uint32 (абсолютные / SMIF-база / номер сектора); связной
   таблицы нет, 98 «кандидатов» оказались повторами байт внутри данных.
8. **Метод FTools** (смещения как абсолютные адреса от фикс-базы, как в Ford IPC) —
   перебор баз `0x60000000/0x08000000/...`; только ложные монотонные
   последовательности, настоящей таблицы нет.
9. **Гипотеза «A5A5A5A5 = контейнер Infineon ResourceGenerator»** — ПРОВЕРЕНА и
   ОПРОВЕРГНУТА. Прогнали официальный `ResourceGenerator.exe` (из SDK Infineon) во всех
   режимах: обычный бинарь (`-b`), append несколько ресурсов (`-a`), partitioning (`-p`).
   Ни в одном режиме `A5A5A5A5` не появляется — вывод всегда начинается с заголовка
   ресурса (nSize, w, h, stride, bpp, flags, colorbits). Значит `A5A5A5A5` в нашем
   дампе — **обёртка/раздел уровня Geely, а не формат Infineon**. Оракул-подход к
   контейнеру (как сработал с RLAD-пикселями) здесь не проходит — контейнер закрытый.
10. **Слепой A8-декод дампа + строгий A8-детектор глифов** — ПРОВЕРЕНО, шум. После
    дореверса A8 (формат глифов) прогнали A8-декодер по всему дампу со строгим
    детектором (валидный cbpc, смесь пустых/чернильных пакетов, доминирующий фон,
    мало уровней alpha). Детектор дал ~112 «кандидатов», НО **визуальная проверка
    всех отрендеренных картинок — снова шум** (серые блоки/полосы, ни одного глифа).
    Причина та же: без таблицы ресурсов декодер стартует не с начала глифа и не знает
    ширину; любой RLAD-подобный поток из случайных байт «разжимается» в фон+шум.
    **Перепроверено после починки построчного декодера, на ОБОИХ чипах (U17 и U18),
    с перебором ширин 12–64 и визуальным просмотром десятков PNG — результат тот же
    шум (горизонтальные полосы = границы блоков).** Метрики (в т.ч. score=1.000)
    оказались ложными: идеальный балл давали ОДНОРОДНЫЕ/пустые зоны, а не глифы.
    Вывод окончательный: даже с идеально работающим A8-декодером слепой поиск в дампах
    невозможен — барьер не в декодере (он верен, тесты 8/8), а в отсутствии привязки
    (таблицы ресурсов). Шумовые PNG-примеры в репозиторий НЕ включены как бесполезные.
11. **Метод «цепочка заголовков» (из анализа SDK Infineon)** — ПРОВЕРЕНО, не найдено.
    Ключевой инсайт из разбора репо Infineon: оглавления-таблицы НЕ существует ни у
    кого — ресурсы просто склеены встык, а «индекс» = хардкод-указатели в коде MCU.
    Правило обхода: найти 20-байт заголовок → прыгнуть на `off + nSize` → там должен
    быть следующий валидный заголовок; цепочка = карта всех ресурсов без таблицы.
    Прогнали по нашему дампу со строгой валидацией (nSize/w/h согласованы,
    `stride == ceil(w*bpp/8)`): **0 цепочек длиной ≥5**. Дополнительно: 0 мест с ТОЧНЫМ
    stride, а обязательное поле RGBA-заголовка `ColorComponentShift = 18 10 08 00`
    встречается лишь 2 раза на 64 МБ (в реальном surface-контейнере были бы сотни).
    Вывод: **стандартных Infineon-контейнеров (ResourceGenerator) в нашем дампе НЕТ.**
    Формат Geely — иной. См. docs/INFINEON_REPOS.md (что даёт открытая экосистема, а
    чего в ней принципиально нет).
12. **Служебные зоны NOR (boot-header / partition table / metadata)** — ПРОВЕРЕНО, пусто.
    По даташиту S25HL512T мелкие 4КБ-сектора (типичное место метаданных) — в начале ИЛИ
    конце чипа; агент рекомендовал искать таблицу в первых и последних 128КБ. Проверили:
    - Начало дампа: только заголовок `A5A5A5A5` + одно 16-бит значение (U17=0x20E0,
      U18=0xE2F0), далее **чистые нули до 0x180000**. Это НЕ таблица разделов — адресов
      ресурсов там нет.
    - Последние 128КБ (top-сектора): **полностью пусты** (0 значимых байт).
    - Значение из заголовка (0x20E0/0xE2F0) — разное у чипов, к границам данных не
      привязывается (проверены множители ×1/×64/×256/×512/×4КБ). Вероятно версия/CRC.
    - Данные обоих чипов идут до ~0x3450000 (≈52 из 64 МБ используется).
    Вывод: таблицы ресурсов в служебных зонах NOR нет — она в прошивке MCU (хардкод
    указатели в область 0x60000000), а MCU-flash под HSM. См. docs/INFINEON_REPOS.md.
13. **Извлечь готовый декодер `RLAD::Decode` из библиотек Infineon** — НЕВОЗМОЖНО.
    Символов `RLAD::Decode`/`BitStream`/`Package` в `libutil_cm7.a` и
    `libbasic_graphics_cm7.a` нет (реализация в закрытом `03_build`, не в публичных `.a`).
    Публичен только заголовок-интерфейс — по нему декодер и реализован в этом проекте.
14. **Дамп внутренней secure-flash MCU через SWD/JTAG — ЗАКРЫТ (подтверждено).** Нужно
    **разделить две операции**, которые здесь долго путали:
    - **(a) Чтение внутренней secure-flash самого MCU** (код + таблица ресурсов @
      `0x10000000`) через debug — **невозможно.** Подтверждено даташитом CYT3DL (§8: mask-ROM
      применяет DAP restrictions при каждом boot; §3.4: интерфейсы отключаются при security)
      + маркировкой чипа `HB` (security ON, RSA-3K) + форумным кейсом на нашем part number
      (закрыта и запись). В эту область залезть нельзя — и практики туда не лезут.
    - **(b) Чтение внешнего NOR через XIP-окно `0x60000000` на РАБОТАЮЩЕЙ приборке** — это НЕ
      дамп secure-flash, а чтение памяти, которую CPU **штатно видит** после boot: SMIF уже
      открыл окно и (если включён) снял AES-слой, MCU разложил разметку («чистые ресурсы
      после приготовлений системы»). Эксперт сообщества залил скрин с **JTAG + UDS**;
      алжирцы так и снимают. Это объясняет, почему наш raw-дамп программатором = шум, а их
      метод работает: разница в **точке съёма**, а не в обходе secure-flash.

    Вывод: таблицу из secure-flash достать нельзя (подтверждено), но она и не нужна —
    рабочий путь (b). Тема (про закрытую запись):
    community.infineon.com/t5/TRAVEO-T2G/Flashing-failed-for-CYT3DLABHBQ1AES-product-development-board/td-p/1010153
    Полный разбор метода — docs/INFINEON_REPOS.md → «Метод алжирцев: JTAG через XIP-окно».
15. **Готовые инструменты Infineon для декода bin→картинка** — НЕ СУЩЕСТВУЮТ публично.
    Вся публичная экосистема односторонняя (image→bin), декод делает аппаратный GPU.
    ResourceGenerator/rlad-encoder — только кодеры; Bin2Text — bin→C-текст (не картинка);
    png_Read — только чтение PNG. Kanzi/Candera/Qt формат Infineon не декодируют.
    Поэтому наш `rlad_decoder.py` (oracle-реконструкция) — единственный рабочий софт-декодер.
16. **MCU при старте сверяет конфигурацию железа/данных — рассогласование НЕОБРАТИМО**
    (данные сообщества, не наш тест). Процессор при включении проверяет согласованность
    (в т.ч. соответствие обоих NOR-чипов) и при несоответствии **пишет флаг во внутреннюю
    память; после этого приборка блокируется даже при возврате исходного состояния**.
    Практический смысл: (а) нельзя запускать приборку с одним чипом из двух или с
    рассогласованными чипами — фатально; (б) метод «проб и ошибок» на живой приборке
    недопустим (цена ошибки = замена узла); (в) заливка должна быть согласованной по обоим
    чипам + области MCU сразу. Это объясняет, почему полный клон чужого дампа (напр.
    мультиязычного донора с иной разметкой в MCU) в лоб = вероятный кирпич. Детали и
    следствия для метода — docs/PRACTICAL_APPROACH.md → «Ограничения метода».

## 2a. Структурный анализ дампов NOR — что проверено на самих байтах (сессия)

Отдельная серия экспериментов **прямо на двух дампах** (U17+U18) — попытки извлечь
графику/разметку структурным анализом, без внешних метаданных. Итог всей серии
единый: **декодер RLAD верен (round-trip 8/8), данные на месте, но нет бит-точного
старта, ширины и формата ресурсов — они в таблице разметки в закрытой flash MCU.**
Ниже — что именно проверено, чтобы это не повторяли. Инструменты сессии:
`tools/sector_map.py` (overview/sectors/raw), `tools/bit_closure.py`.

**Подтверждённые факты о раскладке (полезны для будущего):**
- **Активная зона данных:** `0x180000`–`~0x3450000` (~52 МБ), до `0x180000` — заголовок,
  после `~0x3450000` — пусто (0xFF). Одинаково на обоих чипах.
- **`A5A5A5A5` = маркер КРУПНЫХ секций**, не per-ресурсный заголовок (≈9 на U17, ≈14 на
  U18). Первая секция (`0x0`) — служебная: маркер + 16-бит значение + нули (версия/тип).
- **Два чипа = РАЗНЫЕ данные в едином адресном пространстве.** Ни одного одинакового
  64КБ-блока; данные U18 не находятся в U17. Это НЕ зеркало, НЕ RAID-чередование, НЕ
  конкатенация — просто расширение объёма (каждый ресурс целиком в одном чипе). Прямо
  объясняет «старт с одним чипом = кирпич» (п.16): половина ассетов физически на 2-м чипе.
- **Разметка секций между чипами СОВПАДАЕТ по структуре** (границы секторов почти
  идентичны), а содержимое разное → **адреса/границы разделов захардкожены и постоянны**.
  Практический вывод: **diff двух дампов на РАЗНЫХ ЯЗЫКАХ выделит языковые регионы
  надёжно** (меняется наполнение, не раскладка).
- **Общие фоны/градиенты дублируются на оба чипа** (напр. секции с заголовками
  `5A 9E 64...` и `89 62 A9...` побайтово идентичны на старте 29199 и 15659 байт). Это
  НЕ таблица смещений, а один и тот же RLAD-поток ровной заливки (автокорреляция период
  21 = 1.00). Низкая энтропия «ровных» зон = сжатый фон, а НЕ несжатый атлас.

**Проверенные-и-ОТКЛОНЁННЫЕ подходы (все дали шум/пусто):**
17. **`find_surfaces` (Infineon 20-байт заголовок + RLAD-валидация)** — 0 surface на
    обоих чипах (обёртка Geely, стандартных заголовков нет — см. п.11).
18. **Слепой рендер «ровных»/flat-зон как сырых байтов** (`sector_map.py raw`) — только
    горизонтальные полосы (= построчный сжатый поток), узнаваемого фона/атласа нет.
19. **RAID / чередование двух чипов** (байт, слово16, секциями; + конкатенация) — нет.
    Доказано: 29199 идентичных байт (при striping были бы комплементарны), цельный
    период-21 ресурс ВНУТРИ одного чипа, кросс-корреляция падает в плотных зонах (73→85).
    Энтропия для этого теста бесполезна (перестановка байт не меняет гистограмму).
20. **Локальный TOC в начале `A5`-секции** — нет; сразу после маркера идёт RLAD-поток.
21. **RLD/RLC-кодек по всем секциям** (`rlc_decoder.decode_rld`, bpp 4/8/16, LE/BE) —
    0 срабатываний score>0.6; секции не в RLD-формате (основной кодек — RLAD).
22. **Поиск таблиц XIP-указателей** (`0x60000000`+ и flash-relative) по всему чипу — 11
    «кандидатов», при разборе все = сжатые пиксели, случайно попавшие в диапазон адресов
    (структура `F8 F9 FF 02...` = RLAD-поток, не монотонные смещения). Настоящих таблиц нет.
23. **Слепой перебор ширин + bimodal/stroke-метрики** — ложные пики на пустых зонах и на
    регулярном сжатом потоке (width-резонанс w=16/32 = период потока, не форма глифа).
24. **bit-closure** (`bit_closure.py`): привязка W×H к длине ресурса из diff-границ
    (ресурс L байт должен «замкнуться» ~L·8 бит). 4677/5400 кусков замкнулись — НЕ
    селективно (при малом cbpc замыкается почти любой размер). Кандидаты глифных размеров
    (W≈16–32, H≈24–28, уникальные между чипами) при рендере = снова полосы. Причина:
    diff даёт байтовую границу, но НЕ бит-точный старт RLAD-пакета; ошибка в 1–2 бита
    раскладывает поток в регулярные полосы. Точный старт+ширина — в дескрипторе из MCU.

**Вывод серии:** из двух NOR-дампов ОДНОЙ машины без внешних метаданных осмысленную
графику извлечь нельзя (проверено ~9 способами). Разблокируют только: (1) второй дамп на
ДРУГОМ ЯЗЫКЕ (diff → привязка к строкам, границы уже посчитаны) или (2) штатный S19
flash-файл (несёт область MCU с таблицей разметки). Слепой разбор как направление —
закрыт доказательно.

> ⚠️ **ЧАСТИЧНО ОПРОВЕРГНУТО (2026-07-08) — см. [SUCCESSFUL_FINDINGS.md](SUCCESSFUL_FINDINGS.md).**
> Вся эта серия неявно предполагала, что графика **сжата RLAD**. Но часть графики лежит
> **несжатым сырым A8** (1 байт = 1 пиксель), и её слепой разбор ВОЗМОЖЕН прямо из дампа:
> извлечена библиотека иконок приборки (зона `0x28F4000`, stride 48). Ошибка серии — что
> raw-A8 зону пытались RLAD-декодировать (декодер разжимал несжатое в шум/полосы). Уточнённый
> вывод: «слепой разбор закрыт» верно **только для RLAD-сжатых** ресурсов; для несжатых —
> открыт (детектор: <90 уникальных байт-значений + бимодальность + автокорреляция ширины).

### РЕШЕНО ПОЛНОСТЬЮ: A8-RLAD дореверсен и верифицирован (формат глифов шрифта)
Изначально декодер был только под **RGBA8888**, A8-вывод ResourceGenerator не читал.
Дореверсили A8 oracle-методом (`ResourceGenerator.exe -foA8 -cRLAD`). Формат A8-пакета:
**4 бита cbpc + 8 бит bias + N×cbpc offset** (один канал; N — число пикселей в пакете).

Три уточнения, найденные при верификации (превратили «работает на глифах» в «формат
понят на 100%»):
1. **Упаковка ПОСТРОЧНАЯ.** Пакеты не переходят границу строки; последний пакет строки
   = остаток `width%8`. Раньше декодер читал непрерывно и съезжал на ширинах, не
   кратных 8 (поймано oracle-диффом на 50×40/37×29). Исправлено.
2. **cbpc=8 в A8 — lossless** (в отличие от RGBA, где cbpc=8 давал lossy-dithering).
   Проверено на `[0,255,0,255,...]` — 0 расхождений. lossy-ветки в A8 НЕТ.
3. **Флаг сжатия** (`flags` @+11): 0x00 = несжато (сырые байты), !=0 = RLAD. Декодер
   смотрит на flags.

Верификация: round-trip decode(ResourceGenerator(x))==x на размерах 16×16, 64×64,
50×40, 37×29, 12×12, 13×20, 7×7 (кратные и НЕ кратные 8; сжатые и несжатые) — 0
расхождений. Тесты `a8:glyph` + `a8:rowwise` в `tests/` (8/8 passed). **Формат глифов
приборки (A8) теперь декодируется полностью и без потерь.**

## 3. Опыт коллег с ДРУГИХ авто (что переносимо, что нет)

Изучены реальные успешные локализации приборок. Общая закономерность:
**никто не переводил приборку из голого чип-офф дампа — работали с update-файлом или
дампом с уже готовым переводом**, где ресурсы и таблица лежат вместе.

- **Ford Focus IPC — `github.com/bigunclemax/FTools`** (открытый C++). Ближайший аналог
  по структуре: таблица смещений + атлас + сжатие + CRC. Приём: смещения = абсолютные
  адреса от фикс-базы (`0x02400000`), файл-offset = addr − base. Работали с VBF
  update-файлом, пересчитывали CRC при пересборке. (У нас метод абс-адресов таблицу не
  нашёл — см. п.8 выше.)
- **Chery Tiggo 8 Pro** (кейс на drive2). QNX-приборка. Автор написал свой
  распаковщик/упаковщик update-файла; первая сборка окирпичила приборку → восстановил
  рефлешем. Стратегически самый близкий: получил **прошивку** (update, не чип-офф) →
  распаковал → правил.
- **Mustang S550 — `Deathstars/mustang-s550-ipc-hebrew`**. Там шрифт = обычный
  встроенный TTF (ищется по magic `00 01 00 00`). У нас стандартных TTF нет — глифы в
  RLAD-битмапах.
- **Предупреждение (Ford i.MX8, Kanzi .kzb)**: даже полный внешний дамп не помог —
  `.kzb` там «не реверсится». У нас в этом смысле повезло: формат RLAD мы вскрыли.

## 4. CMA vs BMA — почему приборки Geely разные

Ключевое разделение (со слов эксперта сообщества):

- **CMA (Monjaro, Preface и др.)** — приборка на прикладном процессоре: Linux + Kanzi
  (или новее Unity) + eMMC (дамп ~8 ГБ). Правится **пофайлово**, перевод относительно
  лёгкий. Все скриншоты `.kzb` с читаемыми строками и «kzb-метод» — отсюда.
- **BMA (наш Coolray / Binyue L, SX11-A5)** — голый микроконтроллер TRAVEO II + 128 МБ
  NOR, графика RLAD, прошивка целиком чипа. Пофайловой замены нет. Это и есть причина,
  почему A5 — самая сложная приборка Geely: **крупным игрокам рынка автопереводов её
  перевести не удалось**; единственная известная успешная локализация — у энтузиастов
  из Алжира в коллаборации с одной СНГ-командой (метод не раскрыт).

Вывод: методы с CMA/Kanzi **напрямую не переносятся** на нашу A5.

### Нюансы формата .kzb (справка, если вдруг пригодится на CMA-платформе)
- `.kzb` — архив **без сжатия**; правится скриптом (распаковать→изменить→запаковать)
  или в hex.
- **Критично:** строка-замена должна быть **строго той же длины**, что оригинал, иначе
  структура разваливается (отсюда сокращения в переводах).
- Строки лежат читаемым текстом; есть секция `Locales` (напр. en-US/neutral/zh-CN).
- Инструменты: **KZB Explorer** (GUI), **python-kzb** (распаковка; запаковка не
  проверена сообществом, часть картинок выходит битой).

## 5. Про ADB и дамп ГУ (частая путаница)

- **ADB к приборке невозможен**: приборка — микроконтроллер Infineon, не Android.
  Дамп её внутренней flash — только SWD/JTAG (и почти наверняка заблокирован HSM).
- **ГУ (большой экран, Flyme/Android) и приборка — РАЗНЫЕ устройства.** К ГУ ADB
  подключается, дамп ГУ снять можно, но **ресурсов приборки в дампе ГУ нет**.
- Единственная возможная польза дампа ГУ для нашей задачи — поискать в Android-части,
  **как ГУ общается с приборкой по CAN** (CarService/VHAL, CAN-обработчики, возможный
  механизм обновления приборки). Связь ГУ↔приборка идёт по CAN (по словам разработчика
  сообщества: медиа-текст на приборку шлётся по CAN, обложки альбомов «не влезают в
  CAN-кадр»).

## 3a. Поиск ТЕКСТА в дампах — исчерпывающая серия (сессия 2026-07-08)

Отдельная большая серия: **как хранится текст приборки?** Проверены ЭМПИРИЧЕСКИ (на байтах,
не рассуждениями) все реалистичные способы хранить текст в embedded-устройстве. **Все дали
отрицательный результат.** Вывод: текст = RLAD-сжатые глифы, слепой разбор невозможен —
доказано перебором альтернатив + тестом стабильности декода (см. конец раздела).

Артефакты рендеров этой серии — в `output/` (см. карту папок в конце). Все проверенные
визуально = шум; в репозиторий как находки НЕ идут, оставлены для перепроверки.

### Проверенные-и-ОТКЛОНЁННЫЕ способы хранения текста
25. **ASCII-строки (len>=6, фильтр «словоподобных»)** — 97369 ASCII-ранов, 12786 прошли
    грубый фильтр, но при осмотре ВСЕ = шум сжатых данных (`fffgfg`, `UUUU`, `PDDUTDE`).
    Осмысленных слов — ноль. Читаемого ASCII-текста в дампе нет.
26. **UTF-16LE (в т.ч. китайский CJK-диапазон)** — доля байт с high-byte в CJK-диапазоне
    14.2% РАВНОМЕРНО по всему дампу = статистический шум, не кластеры строк. Нет.
27. **GBK/GB2312 (родная китайская двухбайтовая кодировка)** — 8846 «строк» из валидных
    GBK-пар длиной >=8 символов, НО все = повторы одного иероглифа (`篌篌篌`, `痧痧痧`,
    `氨氨氨`) внутри графических зон (0x87xxxx, 0x2900000). Реальный текст был бы коротким
    (4-15 симв) и РАЗНООБРАЗНЫМ. Это байтовые паттерны графики, не текст. Нет.
28. **Векторный шрифт (TTF/OTF/FreeType/WOFF)** — сигнатуры sfnt(`00010000`)/OTTO/true/
    ttcf/wOFF. `00010000` встретилась 14582 раза, но НИ ОДНА не прошла валидацию как
    настоящий sfnt (numTables + осмысленные теги таблиц cmap/glyf/head). OTTO/true/ttcf/
    wOFF = 0. Встроенного векторного шрифта НЕТ. (Согласуется: приборка — голый MCU без
    движка рендеринга текста из контуров в рантайме.)
29. **Несжатый растровый шрифт A1 (1 бит/пиксель)** — развернули байты в биты (MSB и LSB),
    рендер при ширинах 128–512 px по всему чипу + программный скоринг «текстовой структуры»
    (плотность чернил 8–50%, дисперсия плотности строк, доля пустых строк-разделителей).
    Единственные «структурные» зоны = наша A8-иконочная (0x28F0000), отдельного A1-атласа
    букв НЕТ. Рендеры: `output/a1_hunt/`.
30. **Сжатые текстовые блобы (zlib/gzip/xz/zstd/bzip2/lz4)** — попытка inflate на каждом
    zlib-заголовке (0x78 01/9c/da) → 0 текстовых блобов. Magic gzip(190 ложных)/bzip2(2)/
    xz(0)/zstd(0)/lz4(2) — ни один не распаковался в текст. Нет.
31. **Исполняемый ARM-код со строковыми литералами / таблицей ресурсов в NOR** — ПРОВЕРЕНО,
    кода в NOR НЕТ:
    - Thumb `push{lr}` (xxB5) на чётных смещениях: 0.147% полуслов — РЕЖЕ случайного (~0.39%),
      без локальной кластеризации (у региона кода была бы плотность в разы выше фона).
    - Дизассемблер capstone (ARM Thumb) на плотнейших зонах: только 22–28 из 256 байт
      декодируются, инструкции бессвязны (`stc2l p7`, обрывы). Настоящий код декодировался
      бы почти на 100% связными потоками.
    - Таблицы указателей в XIP(0x60000000)/flash(0x10000000)/RAM(0x28000000): единичные
      короткие run'ы (случайность), связной таблицы нет.
    Вывод: код исполняется из ВНУТРЕННЕЙ flash MCU (0x10000000), NOR = только графика.
    Строки/таблица ресурсов — в закрытом MCU, физически не в NOR.
32. **Таблица строк как индексы глифов (несжатая)** — поиск зон малых чисел со структурой
    (index-like: мало нулей, значения в mid-range, умеренное число уникальных). 4 кандидата,
    все = A8/RLAD-хвосты, не таблицы индексов. Несжатой таблицы строк нет.
33. **Периодические атласы (автокорреляция периода по всему чипу)** — найдена СИЛЬНАЯ
    периодичность, но вся объясняется НЕ атласом несжатых глифов:
    - период 48 @0x2900000 = наши иконки (raw A8) ✅
    - период 21 @0x2320000/0x20A0000 (score 0.97–0.99) = RLAD-ритм РОВНОЙ ЗАЛИВКИ фона
      (записи байт-в-байт идентичны в начале зоны = сжатый градиент, distinct=256 дальше)
    - период 32 @0x880000 = шаг градиента-заливки (доминируют 0xb0/0xb1 попарно)
    - период 168/273 = размер записи RLAD-контейнера в графических зонах
    21 и 273 НЕ кратны степени двойки → это ритм RLAD-упаковки, а НЕ байт-выровненная
    таблица данных. Рендеры: `output/period_probe/`.
34. **A8-RLAD-декодер на 16-байт-выровненных стартах в RLAD-зоне (0x2326000+)** — финальный
    прицельный заход, использующий ВСЕ находки (выравнивание-16 + рабочий A8-декодер +
    скоринг глифов). 2241 «кандидат» по скорингу, НО **тест стабильности их разоблачил**:
    настоящий RLAD-глиф при сдвиге старта на ±1 байт ДОЛЖЕН разрушиться (RLAD
    самосинхронизируется без маркеров), а у кандидатов similarity к базовому декоду
    оставалась 0.80–0.95 при сдвигах −16..+16. Значит декодер выдаёт СТАТИСТИКУ зоны
    (стабильную серую кашу с редкими ink-пикселями), а не реальную структуру данных.
    Это артефакт, не глифы. Рендеры: `output/text_hunt/`.

### Тест стабильности декода — ключевой инструмент вердикта (новое)
Придуман в этой серии, годится для любой будущей проверки «глиф или шум»:
> Декодируем кандидата с точного старта, затем со сдвигом старта на −16..+16 байт.
> Реальный RLAD-ресурс: только delta=0 даёт чистый результат (similarity к другим сдвигам
> резко падает, ~0.3), потому что RLAD start-sensitive. Шум/артефакт: similarity остаётся
> высокой (0.8+) при любом сдвиге, т.к. это свойство статистики зоны, а не данных.
Кандидаты из п.34 показали 0.80–0.95 → артефакт. Порог: real ⇒ similarity(delta≠0) < ~0.4.

### Итог серии 3a
Текст хранится ТОЛЬКО как **RLAD-сжатые глифы**. Это не предположение, а результат
исключения ВСЕХ альтернатив (строки/шрифт-вектор/несжатый растр/архивы/код/индексы/атласы)
+ тест стабильности, доказавший что слепой RLAD-декод даёт кашу без точного старта. Ключ к
тексту — тот же, что к цвету: (1) таблица разметки из MCU (JTAG XIP-окно) даёт точный
старт+ширину каждого глифа, ИЛИ (2) diff двух ЯЗЫКОВЫХ дампов покажет, какие RLAD-регионы =
текст, без декодирования формата. Слепой поиск текста — закрыт доказательно.

### Побочные СТРУКТУРНЫЕ находки серии (полезны для diff/будущего)
- **Выравнивание ресурсов на 16 байт: 100%** (240/240 стартов после нулевых промежутков
  в зоне данных кратны 16). Фундаментальное правило раскладки — старты RLAD не произвольны.
- **Раскладка «острова несжатого A8 в море RLAD»:** несжатые иконки — отдельными островами
  (главный 0x28F4000 + мелкие ~0x87C000, ~0xD84000), между ними и во второй половине —
  RLAD-сжатый цвет/графика. Границы секторов совпадают между чипами → адреса захардкожены.
- **Зона иконок (raw A8) — точно не зашифрована** (читается байт-в-байт, см.
  SUCCESSFUL_FINDINGS §1). Про **RLAD-зоны вопрос ОТКРЫТ.** Измерения: энтропия активной
  зоны 7.09 (не ~8.0 как AES), chi-square далёк от равномерного, 8.6% соседних равных байт
  (AES дал бы 0.39%), 4.6% точных дублей 512-байт блоков (XTS убил бы повторы). ⚠️ Эти
  цифры **не доказывают** отсутствие шифрования RLAD-регионов: chi-square/энтропия не
  отличают сжатие от шифрования, а метрики набраны в основном на ровных/несжатых зонах.
  SMIF on-the-fly AES не исключён — см. INFINEON_REPOS.md.
