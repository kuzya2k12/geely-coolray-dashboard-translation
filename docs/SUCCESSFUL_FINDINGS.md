## 🎯 №1. Библиотека иконок приборки — извлечена из сырого дампа (raw A8)

**Дата:** 2026-07-08. **Статус:** подтверждено, воспроизводимо на обоих чипах (U17 и U18).

### Суть открытия

Часть графики приборки хранится в NOR **НЕ в сжатом RLAD, а СЫРЫМ A8** — то есть
**1 байт = 1 пиксель alpha** (grayscale), без всякого сжатия. Достаточно интерпретировать
байты напрямую как изображение при верной ширине строки (stride) — и появляются чёткие,
распознаваемые иконки.

**Почему это пробивает главный барьер проекта.** Весь прежний корпус исследования (см.
INVESTIGATION_HISTORY.md §2a, ~9+ отклонённых методов) исходил из того, что «слепой разбор
дампа невозможен без таблицы разметки из закрытого MCU». Для сжатых RLAD-зон это по-прежнему
верно. Но **для несжатых raw-A8 зон — НЕТ**: их видно напрямую. Ошибка была в том, что эту
зону пытались RLAD-декодировать (декодер «разжимал» несжатые байты в шум/полосы), вместо того
чтобы прочитать байты как есть.

### Точные параметры находки

| Параметр | Значение |
|---|---|
| Зона иконок (первичная находка) | `0x28F4000 .. 0x2928000` (~208 КБ) — включает шумовой лид-ин в начале |
| Зона ЧИСТЫХ иконок (уточнено в §4) | `0x28F8000 .. 0x2940000` — читаются идеально на w48 |
| Практика | зоны между U17/U18 **чуть различаются** — не жёсткий интервал. Иногда старт удобнее брать с `0x28F4000` (со старым лид-ином, на всякий случай), иногда с `0x28F8000`. Пробуй оба интервала — и старый, и новый |
| Формат | **raw A8** (1 байт = 1 пиксель alpha, несжато) |
| Stride (ширина строки) | **48 пикселей** |
| Разделители иконок | пустые (все-нулевые) строки между ресурсами |
| Извлечено иконок | ~64–68 (U17: 68, U18: 65 — мелкое расхождение = разное число ресурсов на чипах) |
| Воспроизводимость | иконки **продублированы на оба чипа** (общий набор на U17 и U18); границы зоны между чипами слегка сдвинуты, набор — тот же |

### Что распознано на иконках

Полный набор телльтейлов приборной панели Geely:
- 🎵 медиа (музыкальные ноты, несколько размеров/состояний)
- ⏱️ спидометры / тахометры (с часами и без — статистика поездки)
- 🚗 иконки авто (вид спереди/сзади, сплошные и «призрачные» контуры)
- 🚴 велосипед, 🚶 пешеходы (ADAS-предупреждения), 👤 airbag, ремень безопасности
- ⚠️ треугольники-предупреждения, 🔒 замки (закрытый / с X)
- 💡 фары / габариты / дальний свет, 💼 портфель/багажник
- ☕ чашка кофе (усталость водителя), 🅿️ парковка «P»
- 🛢️ маслёнка (масло двигателя), MAX/MIN уровни, 🔑 ключ, ⚙️ сервис-шестерёнка

Многие иконки идут **парами** (активное/неактивное состояние ИЛИ тёмная/светлая тема — в
панораме видны инвертированные, чёрное-на-белом, варианты; согласуется с наличием двух тем,
подтверждённым фото в `photos/`).

### Как воспроизвести

```bash
python tools/extract_raw_a8.py Coolray_25_Dash_U17.bin
```
Создаёт:
- `output/found_glyphs/<dump>_panorama_w48.png` — **большое полотно всей зоны** (видно всё разом)
- `output/found_glyphs/icons_<dump>/ic_NNN_offXXXXXXXX_WxH.png` — отдельные иконки, оффсет в имени

Просканировать весь чип на другие raw-A8 зоны:
```bash
python tools/extract_raw_a8.py Coolray_25_Dash_U17.bin --scan
```

### Метод (как нашли — для повторного применения)

1. **Искать зоны мелких ресурсов с нулевыми промежутками** — так нашёлся кластер около `0x2900000`.
2. **Байтовый профиль как детектор сжатия vs сырья:** сжатый RLAD использует ВСЕ 256 значений
   байта; сырой A8 — лишь несколько уровней alpha. У найденной зоны — **всего 46 уникальных
   значений**, сильно бимодально (≈86% низких = фон + ≈10% высоких = чернила). Это однозначно
   НЕ сжатие.
3. **Ширину (stride) найти автокорреляцией по вертикали:** у настоящего битмапа штрихи
   вертикально непрерывны → верный stride максимизирует схожесть соседних строк. Максимум на
   **w=48** (и w=96=48×2 второй по силе — подтверждает).
4. **Резать по пустым строкам** между иконками.

### Практическое значение

Эти иконки можно **править прямо в дампе**: перерисовал пиксели → записал байты на тот же
оффсет (размер сохраняется, т.к. несжато) → залил по UDS. Для них **НЕ нужен ни RLAD-энкодер,
ни таблица разметки из MCU** — то, что раньше считалось обязательным. Это самый лёгкий путь к
правке графики (ср. цветные RLAD-иконки в PRACTICAL_APPROACH.md → «Вторичная цель»).

### Честные оговорки

- Это **иконки/символы**, не обязательно текстовые глифы шрифта (цифр/букв/иероглифов в этой
  зоне нет). Текст шрифта, возможно, лежит отдельно и может быть сжат RLAD — не подтверждено.
- **Крупные зоны второй половины чипа** (candidate raw-A8 при грубом `--scan`) при рендере дают
  **шум** — они сжаты RLAD, узкий алфавит даёт ложное срабатывание детектора. Не каждая
  «raw-A8-подобная» зона реально глифовая; проверять визуально.
- Начало зоны (`0x28F4000..~0x28FE000`) содержит шумовой лид-ин (хвост сжатого потока перед
  иконками) — на панораме это диагональные полосы в первой колонке.

### Что это меняет в остальных доках

- Опровергает категоричность вывода «слепой разбор невозможен» (INVESTIGATION_HISTORY.md) —
  **для несжатых зон возможен**. Барьер разметки остаётся только для RLAD-сжатых ресурсов.
- Даёт первую реальную опору для diff двух языков: теперь известен минимум один регион с
  прямо-читаемым контентом.

## №2. Структурные находки о раскладке дампа (сессия 2026-07-08)

Не «извлечённая графика», а **подтверждённые правила структуры** — полезны для diff и
будущего разбора. Все получены эмпирически на байтах.

### Выравнивание ресурсов на 16 байт — 100%
Все 240 стартов ресурсов (позиции после нулевых промежутков в зоне данных) кратны **16**.
Старты RLAD не произвольны → любой будущий перебор старта делать ТОЛЬКО по 16-байтным
границам (в 16 раз меньше вариантов).

### Раскладка «острова несжатого A8 в море RLAD»
- Несжатые raw-A8 иконки — отдельными островами: главный `0x28F4000`, мелкие ~`0x87C000`,
  ~`0xD84000`.
- Между ними и вся вторая половина чипа — RLAD-сжатый цвет/графика.
- Границы секторов совпадают между U17/U18 → адреса захардкожены → **diff двух ЯЗЫКОВ
  надёжно выделит языковые регионы** (меняется наполнение, не раскладка).
- Активная зона данных: `0x180000`–`~0x3450000`; первые 1.5 МБ почти пусты (маркер+нули).

### Периодичность (автокорреляция) — вся объяснена, атласа текста НЕТ
- период 48 @`0x2900000` = иконки (raw A8) ✅
- период 21 @`0x2320000` (score 0.97–0.99) = RLAD-ритм ровной ЗАЛИВКИ фона
- период 32 @`0x880000` = шаг градиента-заливки
- период 168/273 = размер записи RLAD-контейнера
Периоды 21/273 не кратны 2^n → это ритм RLAD-упаковки, НЕ байт-выровненная таблица.

### Зона иконок (raw A8) — точно НЕ зашифрована; про RLAD-зоны вопрос ОТКРЫТ
**Достоверно:** raw-A8 иконки читаются напрямую байт-в-байт (см. §1) → эта зона
**гарантированно не зашифрована**. Про остальное (RLAD-сжатые регионы) — **нельзя ни
подтвердить, ни опровергнуть**: без таблицы адресов ни один RLAD-ресурс из дампа не
декодировался, а измерения ниже шифрование **не исключают**.

Ранее здесь стоял категоричный вывод «весь дамп НЕ зашифрован» по измерениям: энтропия
активной зоны 7.09 (не ~8.0 как AES), 8.6% соседних равных байт (AES→0.39%), 4.6% точных
дублей 512-байт блоков (XTS убил бы повторы). ⚠️ **Это НЕ доказательство отсутствия
шифрования всего дампа:** такие метрики отражают в основном несжатые/ровные зоны, а
chi-square/энтропия **не отличают сжатие от шифрования**. SMIF умеет on-the-fly AES —
графика в RLAD-зонах **возможно зашифрована at-rest** (открытый вопрос, см.
[../README.md](../README.md) и INFINEON_REPOS.md → «SMIF on-the-fly AES»).

## Статус: поиск ТЕКСТА — исчерпан, текст = RLAD-глифы (см. INVESTIGATION_HISTORY §3a)

За эту сессию ЭМПИРИЧЕСКИ проверены и ОТКЛОНЕНЫ все способы хранить текст: ASCII/UTF-16/
GBK-строки, векторный шрифт (TTF/OTF), несжатый A8/A1-шрифт, zlib/gzip/xz-блобы, ARM-код
со строками (кода в NOR НЕТ), таблица индексов, периодические атласы, A8-RLAD на
выровненных стартах (разоблачён тестом стабильности декода). **Вывод: текст = RLAD-сжатые
глифы, слепой разбор доказательно невозможен.** Ключ — таблица разметки из MCU (JTAG
XIP-окно) ИЛИ diff двух языковых дампов. Полная серия из 10 методов — в
[INVESTIGATION_HISTORY.md](INVESTIGATION_HISTORY.md) §3a.

**Новый инструмент вердикта — тест стабильности декода:** реальный RLAD-ресурс при сдвиге
старта на ±1 байт разрушается (similarity <~0.4); шум/артефакт остаётся стабильным (0.8+).
Порог отсева ложных «глифов» для будущих проверок.

## №4. Базовая единица иконочной сетки = 24px, stride иконок = 48 (2×24) — сессия 2026-07-09

Уточнение к №1 по итогам детального ручного просмотра + автокорреляции.

### Фундаментальная ячейка = 24 пикселя
Автокорреляция на несжатой иконочной зоне дала ОДНОЗНАЧНЫЙ результат по обеим осям:
- **вертикальная когерентность:** пик на w=24 (0.92), затем кратные 48, 72, 96... (все хуже)
- **горизонтальный период** (шаг элемента): пик на lag=24 (0.91), затем 48, 72...
Соседние ширины (±1: 23, 25) заметно хуже → база именно **24**. Иконки телльтейлов лежат при
stride **48 = 2×24**; более широкие ресурсы — кратные 24 (72, 96, 120...). Это внутренняя
сетка раскладки несжатой графики.

### Точные границы ЧИСТОЙ иконочной зоны
Ручной просмотр колонок (скан `a8/w0048_025B0000.png`) уточнил:
- **чистые иконки: `0x28F8000 .. 0x2940000`** (колонки 35–37 в тайле) — читаются идеально на w48;
- с **`0x2940000`** (колонка 38+) начинается ПЕРЕХОД в RLAD: distinct растёт (44→78→256),
  на w48 «полосит/дублируется», родной ширины уже нет = сжатое. Там глазами ловить нечего.
Финальные галереи: `results/U17_icons_0x28F8000_w48.png` и `results/U18_icons_0x28F8000_w48.png`.

### Ловушка, подтверждённая на практике
«Дублирующиеся иконки строкой» и красивая периодичность в зоне `0x2918000`+ — это АРТЕФАКТ
RLAD-упаковки, НЕ реальные повторяющиеся иконки. Проверка: distinct=123–158 (сжато; у настоящих
иконок 44) + тест стабильности декода 0.86 (шум). Правило-вердикт: **distinct<64 + бимодальность
>0.85 = сырой контент (читается); distinct>120 = RLAD (псевдо-узор, не контент)**.

### Итог по несжатому контенту (окончательно)
Единственная зона с читаемым несжатым контентом = иконки `0x28F8000..0x2940000`. Прочёсаны все
«чистые по distinct» зоны (0x8A0000, 0xC00000, 0x1328000, 0x2320000 и др.) — там заливки/
градиенты/фрагменты, узнаваемых картинок нет. Вывод №1 остаётся: несжато читаются только иконки,
всё прочее (текст, цвет) — RLAD, нужен ключ (JTAG XIP-таблица / diff языков).
